# F&O Database Design — Data Analyst Assignment

## Overview
Relational database for storing and analyzing high-volume Futures & Options (F&O) data from NSE, BSE, and MCX using DuckDB, with a normalized 3NF schema.

## Dataset
- **Source**: [NSE F&O Dataset 3M — Kaggle](https://kaggle.com/datasets/sunnysai12345/nse-future-and-options-dataset-3m)
- 2.5M+ rows, 16 columns: INSTRUMENT, SYMBOL, EXPIRY_DT, STRIKE_PR, OPTION_TYP, OHLC, SETTLE_PR, CONTRACTS, VAL_INLAKH, OPEN_INT, CHG_IN_OI, TIMESTAMP
- Extended with synthetic BSE (SENSEX) and MCX (GOLD, CRUDEOIL) data for cross-exchange analysis

## Schema Design — Why 3NF?

### Normalization Choice: 3NF over Star Schema

| Criterion | 3NF (chosen) | Star Schema |
|-----------|-------------|-------------|
| Redundancy | None — each fact stored once | High — dimensions denormalized into fact table |
| Insert cost | Low — small fact rows | High — wide rows, repeated data |
| Update anomalies | Eliminated via FK constraints | Present in denormalized dims |
| Read performance | Requires JOINs | Faster aggregations (no JOINs) |
| HFT ingestion (10M+/day) | Efficient | Expensive |

**Why not star schema for this use case:**
- Star schema suits read-heavy data warehouses with infrequent writes
- F&O ingestion is write-heavy (daily bulk load + continuous tick data)
- Storing "NSE" or "NIFTY" in every one of 2.5M rows wastes ~40% storage
- FK constraints catch bad data at ingestion time — critical for trading

### 4-Table Schema

```
EXCHANGES ──1:M──> INSTRUMENTS ──1:M──> EXPIRIES ──1:M──> TRADES
```

| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `exchanges` | 3 rows (NSE, BSE, MCX) | exchange_id PK, name, country, currency |
| `instruments` | Unique symbol+type+exchange combos | instrument_id PK, exchange_id FK, symbol, instrument_type |
| `expiries` | Each unique expiry/strike/option_type | expiry_id PK, instrument_id FK, expiry_dt, strike_pr, option_typ |
| `trades` | Daily OHLC/volume per expiry | trade_id PK, expiry_id FK, timestamp, open/high/low/close, settle_pr, contracts, open_int, chg_in_oi |

**Why a separate Expiries table?**
- A single NIFTY symbol can have 200+ strikes × 2 option types × 3 expiries = 1200+ rows per day
- Storing strike_pr + option_typ + expiry_dt in every trade row = massive duplication
- Expiries table normalizes this: each unique (symbol, expiry, strike, option_type) combo stored once

## Scalability for 10M+ Rows HFT Ingestion

1. **Partitioning**: Partition `trades` by `expiry_dt` — queries filtering on a single expiry only scan that partition
2. **BRIN Indexes on timestamp**: `trades.timestamp` is monotonically increasing — BRIN index covers the whole column in tiny space (1/1000th of B-tree size)
3. **Columnar storage** (DuckDB): Only reads columns your query needs — `SELECT symbol, close` doesn't scan all 12 columns
4. **Bulk insert via pandas merge**: Avoids row-by-row Python loops — 2.5M rows inserted in seconds
5. **Vectorized execution** (DuckDB): Processes data in 1024-row batches using SIMD — 10-100x faster than row-at-a-time

## SQL Queries Summary

| Query | Description | Techniques |
|-------|-------------|------------|
| Q1 | Top 10 symbols by OI change | 4-table JOIN, GROUP BY, ABS() |
| Q2 | 7-day rolling volatility | `STDDEV() OVER (ROWS 6 PRECEDING)` |
| Q3 | MCX gold vs NSE index settle_pr | CTE, UNION ALL, cross-exchange |
| Q4 | Option chain with implied volume & PCR | CASE WHEN pivot, NULLIF |
| Q5 | Peak volume last 30 days | `ROW_NUMBER() OVER`, `INTERVAL` |
| Q6 | OI build-up/unwinding classification | CASE logic on OI + price direction |
| Q7 | Max pain strikes per expiry | `RANK() OVER (PARTITION BY expiry)` |
